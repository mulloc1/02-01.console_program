# 콘솔 가계부 평가표 답변 가이드

`평가표.md`의 질문에 대해, 짧고 설득력 있게 답변하기 위한 실전용 정리 문서입니다.

## 사용 방법 (핵심)

- 답변 구조는 항상 **결론 → 구현 방법 → 근거(코드/동작 결과)** 순서로 말한다.
- 기능 질문은 **“된다/안 된다”** 먼저 말하고, 바로 검증 방법을 덧붙인다.
- 설계 질문은 **“왜 이렇게 나눴는지”**를 책임 중심으로 설명한다.
- 확장성 질문은 **현재 한계 + 개선안 + 트레이드오프**까지 말한다.

## 1. 기능 완성도 답변법

### 1-1. 명령어 동작 질문 (add/list/search/summary/expoㄹrt/import/update/delete)

**답변 프레임**

1. 모든 명령이 요구사항대로 동작한다고 결론 제시 
2. 핵심 시나리오 1~2개 설명  
3. 예외 입력 시 오류 처리 방식 설명

**예시 답변**

> 네, 필수 명령은 모두 동작합니다. 예를 들어 `add`로 거래를 저장한 뒤 `list`/`search`에서 즉시 조회 가능하고, `update`/`delete` 결과도 재조회로 검증했습니다. 잘못된 인자 형식이나 누락 입력 시에는 스택트레이스 대신 사용자용 오류 메시지와 해결 힌트를 출력하도록 처리했습니다.

### 1-2. 데이터 유지성 질문 (재실행 후 유지)

**예시 답변**

> 거래/카테고리/예산을 각각 파일로 분리 저장해 프로그램 재실행 후에도 상태가 유지됩니다. 실제로 저장 후 프로세스를 종료하고 다시 실행해 동일 데이터가 조회되는 것을 확인했습니다.

### 1-3. category add/list/remove 질문

**예시 답변**

> 카테고리 추가/조회/삭제를 지원하고, 삭제 시에는 해당 카테고리를 참조 중인 거래가 있는지 먼저 검사해 안전하게 처리합니다. 즉, 무결성을 깨는 삭제는 차단하거나 안내 메시지로 유도합니다.

### 1-4. budget set + summary 질문

**예시 답변**

> `budget set`으로 저장된 예산은 `summary`에서 총 지출 대비 사용률로 계산해 보여주며, 초과 시 경고 상태를 함께 출력하도록 구현했습니다.

### 1-5. CSV import/export 스키마 질문

**예시 답변**

> CSV는 UTF-8, 헤더 포함, 사전에 정의한 컬럼 순서를 기준으로 입출력합니다. export 파일을 다시 import해도 같은 구조로 복원되는 라운드트립 시나리오를 기준으로 검증했습니다.

**UTF-8 검증까지 묻는 경우(추가 답변 포인트)**

- 인코딩 확인은 **파일 자체 확인 + 프로그램 동작 확인** 두 단계로 답변한다.
- 파일 자체 확인: `file -I <csv파일>` 결과에서 `charset=utf-8` 확인
- 프로그램 동작 확인: UTF-8이 아닌 파일 import 시 `[ERROR] CSV 인코딩이 UTF-8 이 아닙니다.` 메시지로 실패하는지 검증
- 코드 근거: `csv_io.py`에서 `encoding="utf-8"`로 열고 `UnicodeDecodeError`를 사용자 오류로 변환
- 테스트 근거: `tests/test_csv_io.py`의 non-UTF-8 입력 실패 케이스

### 1-6. 오류 메시지/종료 코드 질문

**예시 답변**

> 사용자 입력 오류나 파일 오류는 커스텀 예외로 처리해서 사람이 읽을 수 있는 메시지와 해결 방법을 제공합니다. 또한 실패 케이스에서는 종료 코드를 0이 아닌 값으로 반환해 자동화 환경에서도 실패를 감지할 수 있습니다.

## 2. 구조 설계 설명력 답변법

### 2-1. 모듈 분리 이유

**추천 포인트**

- `cli`: 입력 파싱/명령 라우팅
- `services`: 비즈니스 로직
- `utils`/`errors`/`decorators`: 공통 기능, 예외, 횡단 관심사

**예시 답변**

> 모듈은 “입력 처리”, “도메인 로직”, “공통 처리”로 책임을 분리했습니다. 이렇게 나누면 변경 범위가 좁아져 유지보수가 쉬워지고, 테스트도 계층별로 분리할 수 있습니다.

### 2-2. 클래스 책임 경계

**답변 프레임**

1. 클래스 A는 무엇을 책임지는지  
2. 클래스 B는 무엇을 책임지는지  
3. 왜 합치지 않았는지

**예시 답변**

> 예를 들어 거래 관리 클래스는 거래 CRUD와 조회 규칙만 담당하고, 예산 관련 클래스는 예산 설정/집계 판단만 담당합니다. 관심사를 분리해 결합도를 낮추고, 각 클래스를 독립적으로 테스트하기 위해 경계를 나눴습니다.

**현재 프로젝트 구조에 맞춘 보완 답변(중요)**

- 이 프로젝트는 **클래스 + 함수 혼합 설계**다.
- `repository` 계층은 클래스로 분리(`TransactionRepository`, `CategoryRepository`, `BudgetRepository`)해 파일 I/O 책임을 고정했다.
- `service` 계층은 상태 없는 비즈니스 규칙이라 클래스 대신 함수(`add_transaction`, `summarize_month` 등)로 구성했다.
- 데이터 계약은 `models.py`(영속 엔티티)와 `types.py`(필터/요약 DTO)로 분리했다.

**면접/발표용 짧은 답변 예시**

> 저는 책임 경계를 클래스와 함수로 혼합해 설계했습니다. 영속성은 Repository 클래스로 묶어 파일 입출력을 분리했고, 서비스 로직은 상태를 들고 있지 않아 함수형으로 단순하게 유지했습니다. 또한 저장 엔티티(`models`)와 계산/전달 타입(`types`)을 나눠 변경 영향 범위를 줄였습니다.

### 2-3. 파일 기반 update/delete 안전성

**예시 답변**

> 파일 수정은 전체 읽기 후 조건 검증, 변경 반영, 저장 순서로 처리해 중간 상태 노출을 최소화했습니다. 삭제/수정 대상이 없는 경우는 명확한 오류를 반환해 사용자가 결과를 오해하지 않도록 했습니다.

**현재 구현 기준 핵심 근거(원자성)**

- 기존 파일을 직접 덮어쓰지 않고, 같은 디렉터리에 임시 파일을 먼저 만든다.
- 변경 결과를 임시 파일에 끝까지 기록한 뒤 `os.replace`로 기존 파일과 한 번에 교체한다.
- 이 방식은 “성공하면 완전 교체, 실패하면 기존 파일 유지” 모델이라 부분 저장(깨진 상태) 위험을 줄인다.
- 예외가 발생하면 임시 파일을 `unlink`로 정리해 찌꺼기 파일이 남지 않게 처리한다.

**면접/발표용 짧은 답변 예시**

> update/delete 저장은 기존 파일을 바로 수정하지 않고 임시 파일에 먼저 전체 결과를 쓴 뒤 `os.replace`로 교체합니다. 그래서 중간 실패가 나도 기존 파일은 보존되고, 성공 시에만 새 상태로 전환되어 원자성을 확보할 수 있습니다. 실패 시 임시 파일도 삭제해 후처리까지 보장했습니다.

## 3. 파이썬 고급 요소 활용 답변법

### 3-1. 제너레이터 (list/search 스트리밍)

**예시 답변**

> 조회 결과를 제너레이터로 순차 처리해 대량 데이터에서도 한 번에 메모리에 올리지 않도록 했습니다. 특히 `search`처럼 필터 조건이 있는 경우 필요한 레코드만 흘려보낼 수 있어 메모리 효율이 좋아집니다.

### 3-2. 데코레이터 분리 이유

**예시 답변**

> 예외 처리/실행 시간 로깅/입력 검증 같은 공통 기능은 데코레이터로 분리했습니다. 이 방식은 비즈니스 로직과 부가 로직을 분리해 중복을 줄이고, 핵심 함수의 가독성을 높이는 데 효과적입니다.

**클로저 질문이 나올 때 설명 포인트**

- 현재 코드에서 클로저는 복잡한 상태 캡처보다는 **데코레이터를 성립시키는 표준 구조(보일러플레이트)**에 가깝다.
- 내부 `wrapper`가 바깥 `func`를 참조(캡처)해 원본 함수를 호출하고, 앞뒤에 공통 로직을 끼워 넣는다.
- 즉 “클로저 자체가 핵심 기능”이라기보다 “공통 처리 재사용을 가능하게 하는 구조적 장치”라고 설명하면 자연스럽다.

**`@functools.wraps` 관련 꼬리질문 대응**

- 데코레이터의 성립 조건은 `함수 -> 함수`를 받아 `함수`를 반환하는 구조이지, `@functools.wraps` 사용 자체는 필수 조건이 아니다.
- 다만 `@functools.wraps`를 쓰면 원본 함수의 이름/문서문자열/시그니처 메타데이터가 보존되어 디버깅·로그·도구 호환성이 좋아진다.
- 그래서 “없어도 데코레이터는 동작하지만, 실무/유지보수 관점에서 쓰는 것이 권장”이라고 답변하면 된다.

### 3-3. 타입 힌트 실효성

**예시 답변**

> 함수 시그니처에 타입 힌트를 적용해 인자와 반환값 계약을 명확히 했고, 정적 검사나 IDE 자동완성 품질이 개선되었습니다. 덕분에 잘못된 타입 전달 실수를 구현 단계에서 빨리 발견할 수 있었습니다.

## 4. 기술 선택과 확장성 답변법

### 4-1. JSONL vs CSV 선택 근거

**답변 프레임**

1. 두 포맷 장단점 비교  
2. 현재 요구사항에 더 맞는 포맷 선택 이유  
3. 향후 전환 가능성

**예시 답변**

> CSV는 호환성이 좋고 사람이 보기 쉬운 장점이 있지만, 중첩 구조 표현은 제한적입니다. JSONL은 레코드 단위 처리와 확장 필드 추가에 유리해 내부 저장 포맷으로 선택했고, 외부 교환은 CSV export/import로 대응했습니다.

**포맷 차이 요약표 (구현 방식과 분리해서 설명)**

| 비교 항목 | CSV | JSONL |
|------|------|------|
| 구조 표현력 | 행/열 중심의 평면 구조에 강함 | 객체/배열/중첩 구조 표현이 자연스러움 |
| 스키마 유연성 | 헤더 기반이라 타입/컬럼 누락 이슈 대응이 필요함 | 키-값 구조라 필드 의미가 명확하고 확장 필드 추가가 쉬움 |
| 타입 표현 | 기본적으로 문자열 중심, 별도 파싱 필요 | 숫자/문자열/배열 등 타입을 값 자체로 표현 가능 |
| 가독성/호환성 | 엑셀·스프레드시트·BI 도구 호환성이 매우 좋음 | 개발자/시스템 간 교환에 유리하나 일반 사용자 친화성은 상대적으로 낮음 |
| 대용량 처리 관점 | 순차 처리 가능하지만 중첩/타입 처리 로직이 늘어날 수 있음 | 한 줄=한 레코드라 부분 처리, 로그성 데이터, 장애 복구에 유리함 |
| 적합한 용도 | 외부 전달, 수동 편집, 리포트 공유 | 내부 저장, 스트리밍 처리, 스키마 확장성 요구 |

### 4-2. 10만 건 확장 시 병목/개선

**예시 답변**

> 현재 구조의 병목은 파일 전체 스캔 기반 조회와 집계입니다. 개선안으로는 인덱스 캐시, 기간/카테고리 조건 선필터링, 저장소를 SQLite 같은 경량 DB로 전환하는 방법을 고려할 수 있습니다.

### 4-3. 깨진 CSV 행 처리 (부분 성공/롤백/리포트)

**예시 답변**

> 신뢰성을 위해 “부분 성공 + 실패 리포트” 전략을 기본으로 두고, 옵션으로 전체 롤백 모드도 제공하는 방식이 적절합니다. 사용자에게는 성공 건수, 실패 행 번호, 실패 사유를 명확히 제공해 재시도 가능성을 높입니다.

## 발표 직전 체크리스트

- 각 항목당 30초 내 답변 가능하게 2~3문장으로 압축했는가?
- “왜” 질문에 대비해 설계 근거를 책임 분리 관점으로 정리했는가?
- 실패 시나리오(입력 오류, 파일 오류, 깨진 CSV) 답변을 준비했는가?
- 확장성(10만 건) 질문에 병목과 개선안을 같이 말할 수 있는가?

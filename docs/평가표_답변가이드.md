# 콘솔 가계부 평가표 답변 가이드

`평가표.md`의 질문에 대해, 짧고 설득력 있게 답변하기 위한 실전용 정리 문서입니다.

## 사용 방법 (핵심)
- 답변 구조는 항상 **결론 → 구현 방법 → 근거(코드/동작 결과)** 순서로 말한다.
- 기능 질문은 **“된다/안 된다”** 먼저 말하고, 바로 검증 방법을 덧붙인다.
- 설계 질문은 **“왜 이렇게 나눴는지”**를 책임 중심으로 설명한다.
- 확장성 질문은 **현재 한계 + 개선안 + 트레이드오프**까지 말한다.

---

## 1. 기능 완성도 답변법

### 1-1. 명령어 동작 질문 (add/list/search/summary/export/import/update/delete)
**답변 프레임**
1) 모든 명령이 요구사항대로 동작한다고 결론 제시  
2) 핵심 시나리오 1~2개 설명  
3) 예외 입력 시 오류 처리 방식 설명

**예시 답변**
> 네, 필수 명령은 모두 동작합니다. 예를 들어 `add`로 거래를 저장한 뒤 `list`/`search`에서 즉시 조회 가능하고, `update`/`delete` 결과도 재조회로 검증했습니다. 잘못된 인자 형식이나 누락 입력 시에는 스택트레이스 대신 사용자용 오류 메시지와 해결 힌트를 출력하도록 처리했습니다.

### 1-2. 데이터 유지성 질문 (재실행 후 유지)
**예시 답변**
> 거래/카테고리/예산을 각각 파일로 분리 저장해 프로그램 재실행 후에도 상태가 유지됩니다. 실제로 저장 후 프로세스를 종료하고 다시 실행해 동일 데이터가 조회되는 것을 확인했습니다.

### 1-3. category add/list/remove 질문
**예시 답변**
> 카테고리 추가/조회/삭제를 지원하고, 삭제 시에는 해당 카테고리를 참조 중인 거래가 있는지 먼저 검사해 안전하게 처리합니다. 즉, 무결성을 깨는 삭제는 차단하거나 안내 메시지로 유도합니다.

### 1-4. budget set + summary 질문
**예시 답변**
> `budget set`으로 저장된 예산은 `summary`에서 총 지출 대비 사용률로 계산해 보여주며, 초과 시 경고 상태를 함께 출력하도록 구현했습니다.

### 1-5. CSV import/export 스키마 질문
**예시 답변**
> CSV는 UTF-8, 헤더 포함, 사전에 정의한 컬럼 순서를 기준으로 입출력합니다. export 파일을 다시 import해도 같은 구조로 복원되는 라운드트립 시나리오를 기준으로 검증했습니다.

### 1-6. 오류 메시지/종료 코드 질문
**예시 답변**
> 사용자 입력 오류나 파일 오류는 커스텀 예외로 처리해서 사람이 읽을 수 있는 메시지와 해결 방법을 제공합니다. 또한 실패 케이스에서는 종료 코드를 0이 아닌 값으로 반환해 자동화 환경에서도 실패를 감지할 수 있습니다.

## 2. 구조 설계 설명력 답변법

### 2-1. 모듈 분리 이유
**추천 포인트**
- `cli`: 입력 파싱/명령 라우팅
- `services`: 비즈니스 로직
- `utils`/`errors`/`decorators`: 공통 기능, 예외, 횡단 관심사

**예시 답변**
> 모듈은 “입력 처리”, “도메인 로직”, “공통 처리”로 책임을 분리했습니다. 이렇게 나누면 변경 범위가 좁아져 유지보수가 쉬워지고, 테스트도 계층별로 분리할 수 있습니다.

### 2-2. 클래스 책임 경계
**답변 프레임**
1) 클래스 A는 무엇을 책임지는지  
2) 클래스 B는 무엇을 책임지는지  
3) 왜 합치지 않았는지

**예시 답변**
> 예를 들어 거래 관리 클래스는 거래 CRUD와 조회 규칙만 담당하고, 예산 관련 클래스는 예산 설정/집계 판단만 담당합니다. 관심사를 분리해 결합도를 낮추고, 각 클래스를 독립적으로 테스트하기 위해 경계를 나눴습니다.

### 2-3. 파일 기반 update/delete 안전성
**예시 답변**
> 파일 수정은 전체 읽기 후 조건 검증, 변경 반영, 저장 순서로 처리해 중간 상태 노출을 최소화했습니다. 삭제/수정 대상이 없는 경우는 명확한 오류를 반환해 사용자가 결과를 오해하지 않도록 했습니다.

## 3. 파이썬 고급 요소 활용 답변법

### 3-1. 제너레이터 (list/search 스트리밍)
**예시 답변**
> 조회 결과를 제너레이터로 순차 처리해 대량 데이터에서도 한 번에 메모리에 올리지 않도록 했습니다. 특히 `search`처럼 필터 조건이 있는 경우 필요한 레코드만 흘려보낼 수 있어 메모리 효율이 좋아집니다.

### 3-2. 데코레이터 분리 이유
**예시 답변**
> 예외 처리/실행 시간 로깅/입력 검증 같은 공통 기능은 데코레이터로 분리했습니다. 이 방식은 비즈니스 로직과 부가 로직을 분리해 중복을 줄이고, 핵심 함수의 가독성을 높이는 데 효과적입니다.

### 3-3. 타입 힌트 실효성
**예시 답변**
> 함수 시그니처에 타입 힌트를 적용해 인자와 반환값 계약을 명확히 했고, 정적 검사나 IDE 자동완성 품질이 개선되었습니다. 덕분에 잘못된 타입 전달 실수를 구현 단계에서 빨리 발견할 수 있었습니다.

## 4. 기술 선택과 확장성 답변법

### 4-1. JSONL vs CSV 선택 근거
**답변 프레임**
1) 두 포맷 장단점 비교  
2) 현재 요구사항에 더 맞는 포맷 선택 이유  
3) 향후 전환 가능성

**예시 답변**
> CSV는 호환성이 좋고 사람이 보기 쉬운 장점이 있지만, 중첩 구조 표현은 제한적입니다. JSONL은 레코드 단위 처리와 확장 필드 추가에 유리해 내부 저장 포맷으로 선택했고, 외부 교환은 CSV export/import로 대응했습니다.

### 4-2. 10만 건 확장 시 병목/개선
**예시 답변**
> 현재 구조의 병목은 파일 전체 스캔 기반 조회와 집계입니다. 개선안으로는 인덱스 캐시, 기간/카테고리 조건 선필터링, 저장소를 SQLite 같은 경량 DB로 전환하는 방법을 고려할 수 있습니다.

### 4-3. 깨진 CSV 행 처리 (부분 성공/롤백/리포트)
**예시 답변**
> 신뢰성을 위해 “부분 성공 + 실패 리포트” 전략을 기본으로 두고, 옵션으로 전체 롤백 모드도 제공하는 방식이 적절합니다. 사용자에게는 성공 건수, 실패 행 번호, 실패 사유를 명확히 제공해 재시도 가능성을 높입니다.

# 콘솔 가계부 미션

---

## 1. 미션 소개

‘작은 서비스’란 기능이 많은 게 아니라 **예외 상황에서도 데이터가 안전한 것**을 말합니다. 이번에 만드는 건 **콘솔 가계부**인데, 제너레이터 스트리밍, 데코레이터 분리, 타입 힌트까지 구조를 제대로 챙깁니다. 단순한 프로그램이 아니라, **유지보수 가능한 설계**로 완성한다는 관점으로 접근하세요.

이번 미션은 Python으로 **파일 입출력 기반**의 가계부 콘솔 프로그램을 구현하는 과제입니다. 수입과 지출 내역을 단순히 저장하는 수준을 넘어, **수정/삭제, 검색, 월별 요약, 카테고리 관리, 예산 초과 경고**까지 포함한 “작은 서비스” 형태로 완성합니다.

학습자는 터미널에서 명령을 입력해 다음을 구현합니다.

| 명령 | 역할 |
|------|------|
| **add** | 내역 추가 |
| **list** | 목록 조회 |
| **search** | 조건 검색 |
| **summary** | 월별 요약·카테고리 리포트 |
| **budget** | 예산 설정·조회 |
| **category** | 카테고리 관리 |
| **update** | 거래 수정 |
| **delete** | 거래 삭제 |
| **import** | CSV 가져오기 |
| **export** | CSV 내보내기 |

데이터는 프로그램 종료 후에도 유지되도록 **JSONL 또는 CSV** 파일로 영구 저장해야 합니다.

또한 **저장 파일을 한 번에 모두 읽지 않고** 제너레이터로 스트리밍 처리하며, 데코레이터로 공통 기능(**예외 처리, 실행 로그, 실행 시간 측정** 등)을 분리합니다. **타입 힌트**를 적용해 함수와 데이터 구조의 계약을 명확히 하고, **모듈 분리**로 유지보수 가능한 구조를 설계합니다.

## 2. 최종 결과물

다음 **10가지 기능**이 정상 동작하는 애플리케이션 **1개**를 완성한다.

### 2.1 필수 기능 요약

| 기능 | 입력·요청 | 출력·화면 |
|------|-----------|-----------|
| **거래 추가 (add)** | 대화형 입력(날짜, 타입, 카테고리, 금액, 메모/태그) | 저장 성공 메시지 + 생성된 거래 `id` |
| **거래 목록 (list)** | `-limit` 등 옵션 | 최신순 거래 리스트(스트리밍 처리) |
| **거래 검색 (search)** | `-from`/`--to`, `-category`, `-type`, `-q`, `-tag` | 조건에 맞는 거래 리스트(최신순) |
| **월별 요약 (summary)** | `-month YYYY-MM`, `-top N` | 총수입/총지출/잔액 + 카테고리별 지출 TOP N |
| **예산 설정/조회 (budget)** | `budget set --month YYYY-MM --amount <금액>` | 저장 성공 메시지 + `summary`에서 예산 사용률·초과 경고 |
| **카테고리 관리 (category)** | `category add` / `list` / `remove` | 카테고리 목록·추가·삭제 결과(사용 중 카테고리 처리 포함) |
| **거래 수정 (update)** | `-id` 기반(옵션 방식 **또는** 대화형 중 **하나로 고정**) | 수정 성공/실패 메시지(없는 `id` 처리 포함) |
| **거래 삭제 (delete)** | `delete --id <id>` | 삭제 성공/실패 메시지(없는 `id` 처리 포함) |
| **가져오기 (import)** | `import --from <csv>` | 일괄 등록 처리 건수 + CSV 반영 확인 |
| **내보내기 (export)** | `export --out <csv>` + `--month` … 또는 `-from`/`--to` | 처리 건수 + CSV 파일 생성 확인 |

### 2.2 추가 조건(필수 구성)

| 항목 | 요구 |
|------|------|
| **저장 파일 분리** | 데이터는 **3개 이상** 파일로 영구 저장(예: `transactions` / `categories` / `budgets`) |
| **README.md** | 실행 방법, 저장 파일 위치·형식, 주요 명령 예시, import/export **CSV 스키마** 포함 |

## 3. 과제 목표

이 과제를 마친 후, 학습자는 아래를 스스로 설명할 수 있어야 한다.

1. **파일 기반 저장(JSONL/CSV)**으로 데이터를 영구 저장하고, CRUD·검색·요약·입출력을 구현할 수 있다.
2. 콘솔 프로그램을 **클래스/모듈**로 구조화하고, 각 계층(**모델 / 저장소 / 서비스 / CLI**)의 책임을 설명할 수 있다.
3. **`yield` 기반 제너레이터**로 대용량 파일도 스트리밍 처리하는 이유와 동작 방식을 설명할 수 있다.
4. **데코레이터**로 공통 관심사(로그·예외·시간 측정)를 분리한 구조와 이유를 설명할 수 있다.
5. **타입 힌트**를 통해 입출력 계약을 명확히 했을 때 얻는 이점을 실제 코드 예로 설명할 수 있다.

## 4. 기능 요구 사항

다음 요구사항을 **모두** 만족해야 한다.

### 4.1 실행 및 입력 방식

| 항목 | 요구 |
|------|------|
| **실행 예시(권장)** | `python -m budget_app <command> [options]` |
| **도움말** | 모든 명령은 `-help` 옵션으로 사용 방법 출력 |
| **입력 기본** | **대화형**. 예: `add` 실행 시 `input()`으로 날짜·타입·카테고리·금액 등 순차 입력 |
| **옵션 허용(권장)** | `search`, `list`, `summary`, `export`, `import`, `delete`는 옵션 인자 방식도 허용 |
| **update** | **옵션 방식 또는 대화형** 중 하나 선택 가능. **문서에 명확히 고정**(§4.7 참고) |
| **옵션 표기** | 리눅스 관례에 맞게 **`-`로 통일**(예: `-help`, `-limit`, `-from`, `-to`, `-month`) |

### 4.2 데이터 모델

**거래 내역(Transaction)**은 최소 아래 필드를 포함해야 한다.

| 필드 | 설명 |
|------|------|
| `id` | 유일 식별자 |
| `type` | `income` / `expense` |
| `date` | `YYYY-MM-DD` |
| `amount` | 양수 |
| `category` | 카테고리명 |
| `memo` | 선택 |
| `tags` | 선택 |

| 항목 | 요구 |
|------|------|
| **정의 방식** | `dataclass` 또는 그에 준하는 구조 |
| **클래스 수** | **최소 2개 이상**의 클래스 사용(예: `Transaction`, `TransactionRepository`, `BudgetStore`, `CategoryStore`, `BudgetService` 등) |

### 4.3 입력 검증

날짜 형식 오류, 음수/0 금액, 허용되지 않은 `type`, 존재하지 않는 `category` 등은 **재입력 요구** 또는 **오류 메시지 출력**으로 처리한다.

### 4.4 저장 정책

| 항목 | 요구 |
|------|------|
| **포맷** | JSONL **또는** CSV 중 **1개** 선택 |
| **파일 분리** | **3개 이상** 필수: `transactions.<fmt>`, `categories.<fmt>`, `budgets.<fmt>` |
| **저장 위치** | 기본 `./data` 권장. 옵션으로 변경 가능(예: `-data-dir`) |

### 4.5 초기 실행(저장 파일 없을 때)

- 파일이 없으면 **자동 생성**하거나, **초기화 안내 메시지**를 출력해야 한다.
- 카테고리 파일이 비어 있으면 아래 **안 A** 또는 **안 B** 중 하나를 택해 동작을 문서에 명확히 한다.

| 안 | 동작 |
|----|------|
| **A** | 기본 카테고리 자동 생성(예: food, transport, rent, etc) |
| **B** | `category add`를 먼저 하도록 안내하고 `add`를 막음 |

### 4.6 add(거래 추가)

- `add` 실행 후 대화형으로 필드를 입력받아 거래를 저장해야 한다.
- `category`는 **등록된 목록에 존재**해야 한다(없으면 안내 후 재입력·등록 유도).
- 저장 완료 시 **생성된 `id`**를 사용자에게 출력해야 한다.

### 4.7 list(목록 조회)

- **최신순**으로 거래를 출력해야 한다.
- `-limit N` 옵션 지원(**기본값** 제공).
- 파일 전체를 한 번에 로드하지 않고, **제너레이터 기반 스트리밍**으로 구현해야 한다.

### 4.8 update / delete(수정·삭제)

- `delete --id <id>`로 특정 거래를 삭제할 수 있어야 한다.
- 존재하지 않는 `id`는 **“없는 데이터”**로 처리하고 사용자 메시지를 출력해야 한다.
- **`update`**는 아래 **안 A** 또는 **안 B** 중 하나로 **문서에 명확히 고정**해 구현한다.

| 안 | 방식 |
|----|------|
| **A** | 옵션 기반: `update --id <id> [--date …] [--type …] [--category …] [--amount …] [--memo …] [--tags …]` |
| **B** | 대화형: 수정할 필드만 선택·재입력 |

- 파일 기반 저장에서 update/delete는 **전체 재작성 / 임시 파일 / 원자적 교체(권장)** 등 안정성을 고려해야 한다.

### 4.9 search(검색)

조건 기반 검색을 지원해야 한다.

| 옵션 | 의미 |
|------|------|
| `-from`, `-to` | 기간 |
| `-category` | 카테고리 |
| `-type` | 타입 |
| `-q` | 메모 키워드 |
| `-tag` | 태그 |

- 검색 결과는 **최신순**으로 출력해야 한다.
- **제너레이터 기반** 스트리밍을 유지해야 한다.

### 4.10 summary(월별 요약)

- `summary --month YYYY-MM` 입력을 받아 해당 월 요약을 출력해야 한다.
- 출력 항목:
  - 총 수입, 총 지출, 잔액(총수입 − 총지출)
  - 카테고리별 지출 합계 **TOP N**(`-top` 옵션)
- 데이터가 없는 달은 **“데이터 없음”**을 명확히 출력해야 한다.

### 4.11 budget(예산)

- `budget set --month YYYY-MM --amount <금액>`으로 월 예산을 저장해야 한다.
- `summary` 실행 시 예산이 설정되어 있으면:
  - 예산 대비 사용률(%)
  - 초과 여부(**초과 시 경고 문구**)
- 예산 데이터도 **반드시 영구 저장**되어야 한다.

### 4.12 category(카테고리 관리)

- `category add` / `list` / `remove`를 제공해야 한다.
- 카테고리 삭제 시, 해당 카테고리를 사용하는 내역이 있으면:
  - 삭제를 막거나
  - 대체 카테고리를 요구해야 한다.

### 4.13 import / export(가져오기·내보내기)

| 항목 | 요구 |
|------|------|
| **import** | `import --from <csv>`로 거래 일괄 등록 |
| **export** | `export --out <csv>`로 조건에 맞는 거래를 CSV로 저장 |
| **export 조건** | `-month YYYY-MM` **또는** `-from YYYY-MM-DD` + `--to YYYY-MM-DD` 중 **하나 이상 필수** |
| **스키마** | 아래 **CSV 최소 스키마** 고정 |

#### CSV 최소 스키마

| column | required | 설명 |
|--------|----------|------|
| `date` | Y | `YYYY-MM-DD` |
| `type` | Y | `income` / `expense` |
| `category` | Y | 등록된 카테고리 |
| `amount` | Y | 양수 정수 |
| `memo` | N | 문자열 |
| `tags` | N | 쉼표(`,`) 구분 문자열 |

- 공통: **UTF-8**, **헤더 포함**

### 4.14 데코레이터(Decorator)

공통 관심사(예: 예외 처리·로그·시간 측정) 데코레이터를 **1개 이상** 구현하고 실제 적용한다.

### 4.15 예외 처리 및 종료 코드

- 오류는 스택트레이스 대신 **원인 + 해결 힌트**로 출력한다.
- 정상 종료는 `0`, 오류 종료는 **0이 아닌 값**으로 종료한다.

### 4.16 모듈화(구조화)

- 한 파일에 몰아넣지 않고 **최소 3개 이상** 모듈로 분리한다.
- **(권장)** CLI / 서비스 / 저장소(파일 I/O) / 모델(데이터 구조)로 책임을 나눈다.

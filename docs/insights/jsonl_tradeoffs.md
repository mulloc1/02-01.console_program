# JSONL(NDJSON) 선택과 트레이드오프

이 문서는 이번 가계부 과제에서 JSONL을 저장 포맷으로 쓸 때의 **장단**, **한계**, **과제 문맥에서의 적절성**을 정리한다. 대화 중 나온 판단을 코드베이스 맥락(`repositories.py`의 줄 단위 읽기, `services.py`의 `replace_all` 기반 수정/삭제)에 맞춰 기록한다.

## JSONL이 무엇에 유리한가

- **한 줄 = 한 JSON 객체**이므로 **append 한 줄 추가**가 자연스럽다. 새 거래를 `append`만 하는 경로와 잘 맞는다.
- **읽기는 줄 단위로 스트리밍**하기 쉽다. `for line in file` → `json.loads` 패턴이 단순하다.
- **로그·이벤트 스트림**처럼 “계속 쌓이고, 나중에 흘려 읽는” 워크로드와 잘 맞는다.

## 한 줄 깨짐 격리(line-level fault isolation)

- 각 줄을 **독립적으로** 파싱할 수 있다.
- 중간 줄 하나가 `JSONDecodeError`로 깨져도, 구현에 따라 **그 줄만 스킵**하고 다음 줄을 계속 읽을 수 있다(이 프로젝트는 stderr에 `[WARN]` 후 스킵).
- **한 덩어리 JSON**(예: 파일 전체가 하나의 배열)은 중간 문법 오류가 나면 **전체 파싱 실패**로 이어지기 쉽다.

한계:

- 격리는 **“나머지 데이터까지 같이 죽지 않게 한다”**는 의미이지, 깨진 줄의 레코드를 자동 복구하지는 않는다. 그 줄 데이터는 사실상 손실로 본다.

## JSONL이 “쓰기 성능 만능”이 아닌 이유

JSONL 포맷 자체가 **랜덤 업데이트 비용을 없애주지는 않는다.**

- **추가(append)**: 보통 유리하다(파일 끝에 한 줄).
- **특정 `id` 한 건만 수정/삭제**하고, 저장소가 **단일 파일 + 제자리(in-place) 패치 없음**이면:
  - 대상을 찾기 위해 **전체 스캔**이 필요할 수 있고,
  - **원자적 교체**를 위해 **임시 파일에 전체를 다시 쓰는 `replace_all`** 같은 패턴이 흔하다.

즉 **“JSONL이라서 수정이 빨라진다”**기보다, **“append·스트리밍·내결함성 쪽에 맞는다”**에 가깝다.  
랜덤 갱신까지 최적화하려면 **인덱스(id → 줄 번호/오프셋)**, **컴팩션**, **DB** 등 추가 설계가 필요하다.

## `id` 컬럼과 인덱싱

- `id`는 레코드를 **논리적으로 식별**하기에 충분하다.
- 파일 안에서 **물리적 위치**를 바로 알려 주지는 않는다. 그래서 단일 JSONL만 쓰면 **id로 찾을 때 O(n) 스캔**이 기본이 된다.
- **빠른 id 조회**를 원하면 `id → line` / `id → offset` 같은 **보조 인덱스**를 유지해야 하며, 쓰기마다 인덱스를 **일관되게 갱신**해야 한다.

## 이번 과제에서의 위치

과제(subject)는 **JSONL 또는 CSV 중 하나**, **3개 파일 분리**, **제너레이터 기반 읽기** 등을 요구한다.

- **교육 목적·범위·표준 라이브러리만**을 고려하면 JSONL은 **합리적인 선택**이다.
- **실서비스 수준의 빈번한 수정/삭제·대용량 랜덤 액세스**를 기준으로 보면 **트레이드오프가 크다**는 것도 사실이다.

따라서 “과제 요구에는 맞지만, 워크로드에 따라 한계가 분명하다”가 정확한 요약이다.

## 코드베이스에서의 대응 관계

| 경로 | JSONL과의 관계 |
|------|----------------|
| `repositories._iter_jsonl` | 줄 단위 읽기, 손상 줄 스킵 가능 |
| `repositories.append` | append 한 줄 |
| `repositories.replace_all` | 수정/삭제/예산 upsert 등 **전체 재작성**에 사용 |
| `services.update_transaction` / `delete_transaction` | `list(iter_transactions())` 후 `replace_all` — 단일 파일 무인덱스 설계의 자연스러운 비용 |

## 참고: 로그 용도와의 궁합

JSONL은 **애플리케이션 로그·감사 로그·이벤트 수집**처럼 **주로 append하고 줄 단위로 소비**하는 형태와 특히 잘 맞는다. 가계부의 **거래 추가**는 이 패턴과 가깝고, **거래 수정/삭제**는 상대적으로 부담이 큰 편이다.

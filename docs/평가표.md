# 콘솔 가계부 평가표

## 1. 기능 완성도
요구된 기능이 정확히 동작하고, 실행 환경 변화에도 데이터가 신뢰성 있게 유지되는지 평가한다.

### 핵심 확인 항목
- `add / list / search / summary / export / import / update / delete`가 요구사항대로 동작하는가?
- 프로그램 재실행 후에도 거래/카테고리/예산 데이터가 유지되는가? (저장 파일 3개 이상)
- `category add / list / remove`가 정상 동작하는가? (삭제 시 사용 중인 카테고리 처리 포함)
- `budget set`이 저장되며, `summary`에서 예산 사용률/초과 여부가 출력되는가?
- `import / export`가 명시된 CSV 스키마(UTF-8, 헤더, 컬럼)로 동작하는가?
- 잘못된 입력/파일 오류에서 스택트레이스 없이 오류 메시지와 해결 힌트를 출력하는가?
- 오류 상황에서 종료 코드가 0이 아님을 확인할 수 있는가?

## 2. 구조 설계 설명력
코드를 어떤 책임 단위로 나누었는지, 설계 의도를 논리적으로 설명할 수 있는지 평가한다.

### 핵심 확인 항목
- 코드가 3개 이상 모듈로 분리되어 있고, 각 모듈의 책임을 어떻게 나눴는지 설명할 수 있는가?
- 최소 2개 이상의 클래스에 부여한 책임 경계를 어떻게 정했는지 설명할 수 있는가?
- 파일 기반 `update / delete`를 어떻게 안전하게 처리했는지 설명할 수 있는가?

## 3. 파이썬 고급 요소 활용
제너레이터, 데코레이터, 타입 힌트 등 언어 기능을 목적에 맞게 적용하고 근거를 설명할 수 있는지 평가한다.

### 핵심 확인 항목
- `list / search`를 제너레이터로 스트리밍 처리한 방식을 어떻게 구현했고, 왜 유리한지 설명할 수 있는가?
- 데코레이터로 분리한 공통 기능이 무엇이며, 왜 분리가 필요했는지 설명할 수 있는가?
- 타입 힌트를 적용해 얻는 이점을 실제 코드 예로 어떻게 확인했고, 왜 도움이 되는지 설명할 수 있는가?

## 4. 기술 선택과 확장성
현재 선택의 근거와 미래 확장 시 대응 전략을 제시할 수 있는지 평가한다.

### 핵심 확인 항목
- JSONL과 CSV 중 선택한 저장 포맷의 장단점을 비교하고, 왜 그 포맷을 택했는지 근거를 말할 수 있는가?
- 거래가 10만 건으로 늘어난다면, 현재 구조의 병목이 어디이며 어떻게 개선할지 설명할 수 있는가?
- `import` CSV에 일부 깨진 행이 섞이면, 어떻게 처리해 사용자 신뢰를 지킬지(부분 성공/롤백/리포트) 설명할 수 있는가?
